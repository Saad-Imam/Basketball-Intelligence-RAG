# Basketball Intelligence RAG

**Live Demo:** [huggingface.co/spaces/saadimam/Basketball-RAG](https://huggingface.co/spaces/saadimam/Basketball-RAG)

---

## What This Project Does

Basketball is a sport wwhere technical depth is consistently underestimated. A casual fan can follow a game yet struggle to understand why a referee call was made, what the structural difference is between a pick-and-roll and a pick-and-pop, or how a Triangle offense differs from Man-to-Man offense. At the coaching level, answering these questions precisely requires access to dense, technically heavy rulebooks and strategy literature that most people never read.

The inspiration for the system’s framing came from Mind the Game and the older Old Man
and the Three podcasts hosted by JJ Redick. As a basketball fan who started watching the
game in 2020, I had issues with understanding what was happening on the court tactically.
That is when I came across JJ Redick, a 15 year NBA veteran, who had 15-20 mins segments
in his podcast explaining specific plays in clear, understandable language. This system aims
to replicate that quality: answering questions from a casual fan asking “what is a defensive
three-second violation?” and a coach asking  about specific rules/plays with equal precision.

This system closes that gap. It is a Retrieval-Augmented Generation (RAG) question-answering pipeline built on two layers of basketball knowledge:

- **Layer 1 — Official Rulebooks:** The NBA Official Playing Rules (2025–26) and the FIBA Official Basketball Rules 2024 (v10a), processed from PDF with Docling for structure-aware text extraction.
- **Layer 2 — Strategy Encyclopedia:** Scraped from [hoopstudent.com](https://hoopstudent.com), covering the Basketball Glossary, Offense, and Defense category pages with full detail-page content for each term.

A user types a question — anything from *"what is a defensive three-second violation?"* to *"how does a 1-3-1 zone force turnovers in the backcourt?"* — and receives a cited, context-grounded answer generated by LLaMA 3.1 8B, with the retrieved source chunks displayed alongside it.

## Technical Architecture

| Component | Choice | Notes |
|---|---|---|
| Embedding | BGE-M3 (BAAI/bge-m3) | Produces 1024-dim dense + learned sparse vectors in one forward pass |
| Vector Store | Pinecone (serverless) | Single hybrid index, dot-product metric, per-strategy namespaces |
| Retrieval | Dense-only (production) | Two-list RRF + CrossEncoder also implemented; see Branches |
| Re-ranker | cross-encoder/ms-marco-MiniLM-L-6-v2 | Applied over top-10 RRF candidates |
| LLM Generation | LLaMA 3.1 8B via Groq | Strict context-only prompting with inline source citations |
| Evaluation | LLM-as-a-Judge (Gemma 3.1 4B) | Faithfulness (claim extraction + verification) and Relevancy |
| Deployment | Hugging Face Spaces (Docker SDK) | FastAPI backend + custom single-page frontend on port 7860 |
| CI/CD | GitHub Actions | Auto-syncs `main` to the HF Space on every push |

## How the Project Was Built

The project was developed in several sequential stages, each preserved on its own branch:

**Stage 1 — Corpus Preparation**
Docling was used to extract structured text from the two rulebook PDFs, preserving the heading hierarchy (Rule → Section) as chunk metadata. Layer 2 was scraped with a custom scraper and stored as structured JSON. Both layers were noise-filtered and prepended with natural-language context prefixes to improve keyword matching.

**Stage 2 — Embedding and Indexing**
BGE-M3 was chosen over separate dense and sparse encoders because it generates both representations in a single forward pass with learned (rather than frequency-based) sparse weights. Due to a version conflict between Docling and FlagEmbedding's `transformers` pins, pre-processing and embedding were run in separate Kaggle notebook environments (dual T4 GPU) with a JSON checkpoint handoff between them.

**Stage 3 — Retrieval Pipeline**
A four-stage pipeline was implemented: BGE-M3 query encoding, dual Pinecone queries (pure dense and pure sparse), Reciprocal Rank Fusion (k=60), and CrossEncoder re-ranking over the top-10 RRF candidates. MMR (Maximal Marginal Relevance) was also implemented as an optional diversity step between RRF and re-ranking.

**Stage 4 — Evaluation**
A RAGAS-style LLM-as-a-Judge framework was built from scratch: claim extraction returns a JSON array of atomic factual claims; each claim is verified against the retrieved context with a `VERDICT: SUPPORTED / NOT SUPPORTED` response; relevancy is computed as the mean cosine similarity between the original query and three questions generated from the answer. All evaluation results stream to the UI via Server-Sent Events.

**Stage 5 — Ablation Study**
Three chunking strategies (Docling HybridChunker, fixed-size sliding window, semantic percentile-breakpoint) were evaluated against up to three retrieval modes (hybrid, dense-only, sparse-only) on a fixed 10-query test set. Results identified semantic chunking with dense-only retrieval as the winning configuration on both faithfulness (98.3%) and relevancy (0.760), which was then merged into `main`.

## Branches

Each branch captures a distinct strategy. The `main` branch reflects the winning configuration (semantic chunking, dense-only retrieval).

| Branch | Chunking Strategy | Retrieval Mode | Description |
|---|---|---|---|
| `main` | Semantic (percentile-breakpoint) | Dense-only | **Production version.** Merged from `semantic` after ablation study. |
| `semantic` | Semantic (percentile-breakpoint) | Hybrid / Dense / Sparse | Full ablation of all three retrieval modes under semantic chunking. |
| `v1.2` | Docling HybridChunker | Hybrid + Dense | Baseline implementation with document-structure-aware chunking. Also contains V1.1 prompt and judge model improvements. |
| `fixed-chunking` | Fixed-size sliding window (512 tokens) | Hybrid | Fixed-window ablation; worst-performing strategy. |

To switch strategies locally, check out the relevant branch.

## Key Design Decisions and Findings

**Semantic chunking outperforms structure-based and fixed-window chunking.** The percentile-breakpoint algorithm groups sentences by topic shift, producing chunks that are densely focused on a single concept. This is the dominant performance variable — the step change in faithfulness between strategies (~10–15 percentage points) is far larger than any difference introduced by changing the retrieval mode.

**Dense-only retrieval outperforms hybrid for this corpus.** Each chunk is prepended with an explicit context prefix (league name, rule number, term name), which means the BGE-M3 dense vector already captures strong keyword signal. The sparse retrieval leg provides little marginal gain while admitting noise, and removing it improves both faithfulness and relevancy.

**Human evaluation caught a failure mode invisible to automated metrics.** A comparative query asking to choose between the 1-2-1-1 and 1-2-2 press defences scored 83% faithfulness and 0.676 relevancy — values that look acceptable — yet the system had failed to retrieve any 1-2-2 chunks and could not make the comparison. This underscores the need for at least a small set of manually designed adversarial queries alongside RAGAS-style automated evaluation.
